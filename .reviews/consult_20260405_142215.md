**Question**: Android-specific bug: Expo Go + expo-av + expo-speech. After expo-speech TTS plays a translation, the next attempt to record with expo-av throws 'streaming not supported while recording'. This is on ANDROID, not iOS. The HIGH_QUALITY preset uses MPEG_4/AAC. What causes this on Android specifically, and what is the correct fix? Current code uses Audio.setAudioModeAsync with shouldDuckAndroid and playThroughEarpieceAndroid. We also call Speech.stop() and wait 300ms before recording. Still fails.

---

Cause on Android: **`expo-speech` leaves the Android audio stack in a playback/streaming state, and your recorder preset is trying to use an output format/encoder combo that Android treats as unsupported for recording in that state in Expo Go.** The error text is misleading, but the real issue is not your 300ms delay or `Speech.stop()` timing.

## Most likely root cause
`Audio.RecordingOptionsPresets.HIGH_QUALITY` on Android uses:

- `outputFormat: MPEG_4`
- `audioEncoder: AAC`

That preset is fine in many cases, but on **Android + Expo Go**, after `expo-speech` has used TTS/audio focus, starting a new `MediaRecorder` with **MPEG_4/AAC** is the combo that commonly fails and surfaces as:

- `"streaming not supported while recording"`

This is Android-specific because:

1. **Android TTS uses native audio focus / stream routing differently than iOS**
2. **Expo Go shares one host app audio environment**, so `expo-speech` and `expo-av` interact more tightly than in a standalone build
3. **Android MediaRecorder is stricter** about valid recorder state / format transitions than iOS AVAudioSession

So the bug is **not primarily that you didn’t wait long enough**. The wait can help sometimes, but it does not fix the underlying Android recorder format/session incompatibility.

---

## Correct fix
On Android, **do not use `HIGH_QUALITY` (MPEG_4/AAC) for recording here**. Use an Android-specific recording config with:

- `outputFormat: THREE_GPP`
- `audioEncoder: AMR_NB` or `AMR_WB`

This avoids the Android MediaRecorder failure after TTS.

### Recommended Android recording options
Use platform-specific options instead of the preset.

## Copy-paste fix

### `services/audio.ts`
```ts
import { Platform } from 'react-native';
import { Audio } from 'expo-av';
import * as FileSystem from 'expo-file-system/legacy';

let recording: Audio.Recording | null = null;

async function setRecordingMode() {
  await Audio.setAudioModeAsync({
    allowsRecordingIOS: true,
    playsInSilentModeIOS: true,
    staysActiveInBackground: false,
    shouldDuckAndroid: true,
    playThroughEarpieceAndroid: false,
  });
}

async function setPlaybackMode() {
  await Audio.setAudioModeAsync({
    allowsRecordingIOS: false,
    playsInSilentModeIOS: true,
    staysActiveInBackground: false,
    shouldDuckAndroid: true,
    playThroughEarpieceAndroid: false,
  });
}

const RECORDING_OPTIONS: Audio.RecordingOptions = {
  android: {
    extension: '.3gp',
    outputFormat: Audio.AndroidOutputFormat.THREE_GPP,
    audioEncoder: Audio.AndroidAudioEncoder.AMR_NB,
    sampleRate: 16000,
    numberOfChannels: 1,
    bitRate: 12200,
  },
  ios: {
    extension: '.m4a',
    outputFormat: Audio.IOSOutputFormat.MPEG4AAC,
    audioQuality: Audio.IOSAudioQuality.HIGH,
    sampleRate: 44100,
    numberOfChannels: 1,
    bitRate: 128000,
    linearPCMBitDepth: 16,
    linearPCMIsBigEndian: false,
    linearPCMIsFloat: false,
  },
  web: {},
};

async function startNativeRecording(retries = 3): Promise<void> {
  const { status } = await Audio.requestPermissionsAsync();
  if (status !== 'granted') {
    throw new Error('Microphone permission is required to record audio');
  }

  if (recording) {
    try {
      await recording.stopAndUnloadAsync();
    } catch {}
    recording = null;
  }

  for (let attempt = 0; attempt < retries; attempt++) {
    try {
      await setPlaybackMode();
      await setRecordingMode();

      const newRecording = new Audio.Recording();
      await newRecording.prepareToRecordAsync(RECORDING_OPTIONS);
      await newRecording.startAsync();
      recording = newRecording;
      return;
    } catch (err) {
      if (attempt < retries - 1) {
        await new Promise((r) => setTimeout(r, 300 * (attempt + 1)));
      } else {
        throw err;
      }
    }
  }
}

async function stopNativeRecording(): Promise<{ audio: string; mimeType: string }> {
  if (!recording) {
    throw new Error('No recording in progress');
  }

  const current = recording;
  recording = null;

  await current.stopAndUnloadAsync();
  await setPlaybackMode();

  const uri = current.getURI();
  if (!uri) {
    throw new Error('Recording failed — no file URI');
  }

  const base64 = await FileSystem.readAsStringAsync(uri, {
    encoding: 'base64',
  });

  const mimeType = Platform.OS === 'android' ? 'audio/3gpp' : 'audio/mp4';
  return { audio: base64, mimeType };
}

export async function startRecording(): Promise<void> {
  if (Platform.OS === 'web') {
    return startWebRecording();
  }
  return startNativeRecording();
}

export async function stopRecording(): Promise<{ audio: string; mimeType: string }> {
  if (Platform.OS === 'web') {
    const audio = await stopWebRecording();
    return { audio, mimeType: 'audio/wav' };
  }
  return stopNativeRecording();
}
```

## Why this fixes it
Because on Android, `THREE_GPP + AMR_*` uses the recorder path that remains reliable after TTS/audio focus changes. `MPEG_4 + AAC` is the problematic path here.

---

## Secondary issue in your code
You currently return:

```ts
return { audio, mimeType: 'audio/mp4' };
```

for all native platforms.

That is wrong once Android records `.3gp`. You must return:

- Android: `audio/3gpp`
- iOS: `audio/mp4`

Otherwise your backend may mis-handle the file even after recording succeeds.

## What is *not* the real fix

- `Speech.stop()`
- waiting 300ms
- calling `setPlaybackMode()` before `setRecordingMode()`
- tweaking `shouldDuckAndroid`
- tweaking `playThroughEarpieceAndroid`

Those may reduce timing issues, but they do **not** solve the Android recorder format incompatibility.
