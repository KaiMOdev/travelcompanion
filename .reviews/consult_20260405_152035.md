# GPT Consult — architect

**Model**: gpt-5.3-chat-latest
**Date**: 2026-04-05 15:20

---

This is a solid feature, but there are a bunch of architectural cracks that will bite you pretty quickly as usage grows. I’ll go layer by layer and be blunt.

---

## 1. DestinationContext design is too heavy (and leaky)

You’re turning `useDestination` into a global context **that fetches data as a side effect of setting state**. That’s risky.

### Problems

- **Tight coupling of state + data fetching**
  - `setDestination()` triggers:
    - AsyncStorage write
    - phrases fetch
    - tips fetch
  - Now every consumer of destination implicitly depends on those side effects

- **Over-fetching**
  - Culture screen doesn’t need phrases/tips immediately
  - Travel screen may not need tips at all anymore
  - Yet both get fetched on every destination change

- **No request cancellation**
  - Rapid destination switching → race conditions
  - Late responses overwrite newer state

- **Global loading/error state is wrong abstraction**
  - One `isLoading` / `error` for:
    - phrases
    - tips
    - future culture categories?
  - UI will become inconsistent fast

### Fix

Split responsibilities:

```ts
// Context = ONLY selection + persistence
type DestinationContextValue = {
  destination: string | null;
  setDestination: (code: string) => Promise<void>;
  loadSaved: () => Promise<void>;
};
```

Then:

- `usePhrases(destination)`
- `useTips(destination)`
- `useCulture(destination, category)`

Each hook:
- owns its own loading/error/cache
- cancels stale requests

## 2. Missing client-side caching strategy (big gap)

Server has cache. Client does not.

### Problem

Switching chips will:
- trigger network every time
- even for already-loaded categories

Bad UX + wasted API calls.

### Fix

Add per-destination + per-category cache in `useCulture`:

```ts
const cache = useRef(new Map<string, CultureEntry[]>());

const key = `${destination}:${category}`;
```

Or better: use React Query / TanStack Query:

```ts
useQuery({
  queryKey: ['culture', destination, category],
  queryFn: () => fetchCultureCategory(destination, category),
  staleTime: 24 * 60 * 60 * 1000,
});
```

This instantly fixes:
- caching
- deduping
- loading states per category
- retries

## 3. API design is fragile and inconsistent

### 3.1 Missing endpoint implementation

Spec defines:

```
GET /destination/:code/culture/:category
```

…but it does not exist in `server/index.ts`.

So right now the spec is incomplete.

---

### 3.2 No validation of `category`

You must whitelist categories server-side:

```ts
const VALID_CATEGORIES = new Set([
  'dos-donts', 'gestures', 'food', 'tipping', 'temple', 'numbers'
]);

if (!VALID_CATEGORIES.has(category)) {
  return res.status(400).json({ error: 'Invalid category' });
}
```

Otherwise:
- prompt injection via URL
- unexpected Gemini prompts
- cache pollution

---

### 3.3 Response schema is not enforced

You trust Gemini to return:

```ts
CultureEntry[]
```

But you **don’t validate anything**.

You will get:
- missing fields
- wrong shapes
- hallucinated keys

### Fix

Validate before caching:

```ts
function isValidCultureEntry(obj: any): boolean {
  return (
    typeof obj.id === 'string' &&
    typeof obj.title === 'string' &&
    typeof obj.body === 'string'
  );
}
```

Reject bad responses instead of caching garbage.

---

### 3.4 Cache structure not defined clearly

Spec says:

```
cultureCache: Map<string, Map<string, {...}>>
```

Issues:
- nested Map is unnecessary complexity
- hard to serialize if you ever move to Redis

### Simpler:

```ts
const cultureCache = new Map<
  string, // `${code}:${category}`
  { data: CultureEntry[]; timestamp: number }
>();
```

## 4. Data model inconsistencies

### 4.1 You now have 3 different content types

- `Phrase`
- `CulturalTip`
- `CultureEntry`

This is already drifting.

### Problem

UI now has to handle:
- different shapes
- different rendering logic
- different TTS logic

### Fix (strong recommendation)

Unify everything:

```ts
type CultureItem = {
  id: string;
  type: 'phrase' | 'tip' | 'generic';
  title: string;
  body?: string;

  speakable?: string;
  romanized?: string;

  metadata?: {
    category?: string;
  };
};
```

Then your UI becomes dead simple:
- one list
- one card renderer

---

### 4.2 “numbers” is structurally different

Numbers:
- require ordering (1–10)
- are not really “cards”
- need tighter formatting

Your current model treats them like generic entries → UX will feel off.

---

## 5. UX edge cases not handled

### 5.1 Category switching race condition

User taps chips quickly:

- requests overlap
- responses resolve out of order
- wrong content flashes

Fix:
- AbortController or React Query

---

### 5.2 Destination switch mid-load

- Old category data may render briefly
- No clearing strategy defined

Fix:
- reset cache or key by destination
- show skeleton instead of stale content

---

### 5.3 Offline mode

Currently:
- no fallback
- no cache persistence

Result:
- Culture tab becomes useless offline

At minimum:
- persist last successful responses (AsyncStorage)

---

### 5.4 Empty / low-quality AI responses

You don’t handle:
- short responses
- duplicate items
- culturally wrong info

At least:
- enforce min length
- dedupe titles

---

## 6. TTS issues

### Problem

You assume:

```
speakable = native script
```

But:
- some TTS engines fail on certain scripts
- numbers may need special formatting
- food names might be incomplete phrases

### Fix

Add:

```ts
ttsLang?: string;
```

And explicitly pass language to TTS engine.

---

## 7. Performance concerns

### 7.1 No pagination / chunking

Right now:
- 8 + 8 + 6 + 5 + 5 + 10 items per category
- fine today

But:
- future expansion = heavy payloads

Consider:
- lazy loading per category (you already do implicitly)

---

### 7.2 Re-render pressure

If context holds:
- phrases
- tips
- destination
- loading

Every change re-renders both tabs.

Split contexts or use selectors.

---

## 8. Prompt design risks

Your prompts are:

- vague about formatting
- inconsistent across categories

Example:

> "Include speakable field"

Gemini may:
- forget it
- rename it
- misformat it

### Fix

Be strict:

```
Return JSON array ONLY.
Each item MUST follow this exact schema:

{
  "id": string,
  "title": string,
  "body": string,
  "countryCode": string,
  "speakable": string | null,
  "romanized": string | null
}
```

---

## 9. Missing analytics hooks

You’re adding:
- categories
- chips
- new tab

But no tracking for:
- which categories are used
- drop-off
- TTS usage

You’re flying blind on product decisions.

---

## 10. Minor but real issues

- “Temple” category is culturally narrow → rename to `sacred-sites`
- Country ≠ language (e.g., India, Switzerland) → numbers prompt may be wrong
- No rate limiting on new endpoint → easy abuse
- No timeout handling for Gemini calls

---

## What I would change (concrete)

If I were refactoring this:

1. **Keep DestinationContext minimal**
2. **Adopt React Query for all fetching**
3. **Unify content model**
4. **Add strict server validation**
5. **Flatten cache key**
6. **Add abort/cancellation**
7. **Persist client cache (optional but high value)**

---

## Bottom line

Good product move, but:

- state management is too coupled
- API layer is under-specified and unsafe
- client caching is missing (biggest UX flaw)
- data model is already fragmenting

If you ship as-is, it’ll work for demos—but will get messy fast as you add more cultural content or scale usage.
