**[BUGS & CORRECTNESS] severity: medium**  
`voxlingo/components/OfflineBanner.tsx:10` — `isConnected` can be `null`, causing the banner to show before connectivity is actually known
> `NetInfoState.isConnected` is nullable during initial resolution on some platforms. `!state.isConnected` treats `null` as offline, so users may briefly see a false offline warning on app start or screen mount.
> Prefer checking `state.isConnected === false` (or combining with `isInternetReachable === false` if you want actual internet reachability rather than just network attachment).
> ```ts
> useEffect(() => {
>   const unsubscribe = NetInfo.addEventListener((state) => {
>     setIsOffline(state.isConnected === false);
>   });
>   return unsubscribe;
> }, []);
> ```

**[DESIGN] severity: medium**  
`voxlingo/app/(tabs)/camera.tsx:209` and `voxlingo/app/(tabs)/index.tsx:149` — each screen creates its own NetInfo subscription
> Mounting `OfflineBanner` separately in multiple tabs means each mounted screen adds its own network listener. This is not catastrophic, but it duplicates subscriptions and spreads connectivity logic across screens.
> Consider lifting connectivity state into a shared provider/hook and rendering the banner once at a layout level, especially since this appears to be app-wide status.
> ```ts
> // e.g. app-level hook/provider
> const useOfflineStatus = () => {
>   const [isOffline, setIsOffline] = useState(false);
>   useEffect(() => NetInfo.addEventListener(s => setIsOffline(s.isConnected === false)), []);
>   return isOffline;
> };
> ```
> Then render a single banner in the tabs layout or pass the shared state down.

**Summary**: The change is small and useful, but the connectivity check has an initial-state correctness issue, and the current placement duplicates global network subscriptions across screens. Fixing those would make this safer and cleaner.
