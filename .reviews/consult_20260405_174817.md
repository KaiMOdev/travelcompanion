## 🔴 Build-breaking / runtime issues

### 1. **`cultureCache` used before declaration (Task 1, Step 3)**

You explicitly note:

> “Move the declaration … up before these helpers”

But the plan does **not show the actual new placement**, and the inserted code **uses `cultureCache` immediately**:

```ts
cultureCache.set(cacheKey, cached);
```

If the implementer forgets to move it (very likely), this will crash with:

```
ReferenceError: Cannot access 'cultureCache' before initialization
```

✅ Fix: The plan must include the exact new location, e.g.:

```ts
const cultureCache = new Map<string, { data: unknown[]; timestamp: number }>();
```

placed **above** the cache helper block.

---

### 2. **Cache key mismatch bug (Task 1, Step 4 vs existing code)**

You change:

```ts
const cacheKey = `${code}-${category}`;
```

But the original code uses:

```ts
const cacheKey = `${code}:${category}`;
```

And **Step 3 startup loader reads filenames → uses `file.replace('.json', '')`**, meaning keys will be:

```
JP-food
```

✅ That part is consistent.

❌ BUT: you did **not remove or update all other usages** of the old key format.

Specifically:
- Any existing in-memory entries created before this change (during same runtime) will use `:` keys
- After deploy, mixed keys can exist in memory if server hot-reloads

✅ Fix:
- Either standardize everywhere (preferred ✅)
- Or explicitly clear old cache / ignore `:` keys

### 3. **Missing imports in `culture.tsx` (Task 3)**

You add:

```ts
const cultureListRef = useRef<FlatList>(null);
```

But:

❌ `useRef` is not imported originally  
Plan says “add it” ✅ good

❌ BUT `FlatList` type is used generically:

```ts
useRef<FlatList>(null);
```

This is **incorrect typing** in React Native.

Should be:

```ts
useRef<FlatList<any>>(null)
```

or better:

```ts
useRef<FlatList<CultureEntry>>(null)
```

Otherwise TypeScript may error depending on config.

## ⚠️ Logic / race condition issues

### 7. **AsyncStorage vs fetch race (Task 2)**

Flow:

1. Read AsyncStorage
2. If valid → `setAllEntries` and `return`
3. Else → fetch

✅ Good

BUT race exists:

- If AsyncStorage is slow
- Fetch starts before storage resolves (because async IIFE)

Actually you **await AsyncStorage before fetch**, so it's safe ✅

However:

### 8. **Double state overwrite edge case**

If:

- AsyncStorage resolves
- THEN fetch already started (due to timing)

You can get:

```
setAllEntries(cached)
then
setAllEntries(fetched)
```

✅ This is acceptable (fresh beats cache)

Not a bug, just worth noting.

---

### 9. **Memory cache ignores TTL (Task 2)**

```ts
const memCached = memCacheRef.current.get(key);
if (memCached) {
  setAllEntries(memCached);
  return;
}
```

❌ No TTL check

So stale data can live forever in memory even if AsyncStorage would invalidate it.

✅ Fix:

Store `{data, timestamp}` in memory too, or remove mem cache entirely.

## ⚠️ Server cache concerns

### 10. **Blocking fs calls in request path**

You use:

```ts
fs.readFileSync
fs.writeFileSync
fs.readdirSync
```

✅ For low traffic, fine

⚠️ But:
- These block the event loop
- Especially risky inside request handler (`readCacheFile`)

✅ Acceptable for MVP, but should be noted.

---

### 11. **No file write concurrency protection**

If 2 requests hit same uncached key:

- both generate
- both write file

✅ Not corrupt (last write wins)
⚠️ But wasted API calls

(Optional improvement: in-flight promise deduping)
