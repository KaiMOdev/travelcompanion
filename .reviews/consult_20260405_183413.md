# GPT Review — Google Play Publishing

---

Here’s a practical review of the spec and implementation plan, focused on what will break, what’s missing, and what should be corrected before trying to publish.

---

# Executive summary

The spec/plan is a solid start, but it has several important gaps:

1. **Security is not fully addressed**
   - `EXPO_PUBLIC_*` secrets are being treated like secure secrets when they are not.
   - API keys exposed in git history need more than rotation; app-side keys must be restricted.
   - Some keys listed should never live in the mobile app at all.

2. **Play Store compliance is incomplete / partly inaccurate**
   - Internal testing is not always “no review.”
   - Data safety and permission declarations are oversimplified.
   - If location is declared in manifest but not truly required, this increases rejection risk.

3. **Build failure risks remain**
   - `app.config.ts` fallback to local IP is dangerous for production.
   - `expo-router` is assumed in config without verifying dependency/plugin presence.
   - `newArchEnabled: true` may introduce unnecessary instability if not already validated.
   - No check for Android package name uniqueness / ownership.
   - No explicit validation of assets, permissions, or plugin compatibility.

4. **There are gaps between spec and plan**
   - Spec says dynamic config should replace static config behavior, but plan still hardcodes most values in `app.json`.
   - Spec mentions privacy policy, service account, OTA updates, and backend hosting constraints, but plan doesn’t fully operationalize them.
   - Spec says “keep existing permissions config,” but plan adds location permissions without proving the app needs them.

5. **Release readiness is overstated**
   - There is no explicit production backend readiness gate.
   - No crash/ANR, app signing, package visibility, or permission prompt testing checklist.
   - No Play Console policy declarations for microphone/camera/location prominence and in-app disclosure review.

# Detailed review

## 1) Missing steps

## A. Missing dependency/config validation before EAS build
Before creating builds, you should explicitly run:

- `npx expo-doctor`
- `npx expo install --check`
- `npx expo config --type public`
- `eas build:configure` if needed
- verify `expo-router` is installed if listed in plugins
- verify all Expo plugins used by the app are compatible with SDK 54

**Why this matters:** EAS builds often fail due to config/plugin mismatches that local Metro doesn’t catch.

**Action:** Add a pre-build validation task:
```bash
cd voxlingo
npx expo-doctor
npx expo install --check
npx expo config --type public
```

---

## B. Missing Android permissions review
The plan blindly adds:

- `CAMERA`
- `RECORD_AUDIO`
- `ACCESS_FINE_LOCATION`
- `ACCESS_COARSE_LOCATION`

But there is no audit of whether the app actually uses location on Android production builds.

**Why this matters:** unnecessary sensitive permissions increase Play review scrutiny and user distrust.

**Action:**  
Only declare permissions that are truly required. If location is optional or not implemented yet, remove it now.

If using Expo config:
```json
"android": {
  "permissions": ["CAMERA", "RECORD_AUDIO"]
}
```

If some library auto-adds permissions, use `blockedPermissions` where appropriate.

---

## C. Missing Android permission rationale / in-app disclosure review
For microphone, camera, and especially location, Play may expect:
- accurate store disclosure
- in-app explanation before system prompt
- feature-linked permission request timing

**Action:** Add a QA step:
- verify permission prompts occur only when user initiates feature
- verify there is an in-app explanation screen/modal before requesting permission
- verify denial path does not crash app

---

## D. Missing app signing / keystore ownership decision
The plan assumes EAS-managed credentials, but does not explicitly state:
- whether Expo manages the Android keystore
- whether the team will download and securely back up the keystore
- who owns Play App Signing and upload key recovery process

**Action:** Add:
- choose **Expo-managed credentials** or manual keystore
- if Expo-managed, run credential setup and document owner access
- back up upload keystore metadata securely

---

## E. Missing package name verification
`com.voxlingo.app` is used as placeholder, but there is no step to verify:
- it is available
- it matches company/domain ownership strategy
- it won’t conflict with future iOS naming / branding

**Action:** Before first build, decide final package:
- preferably based on owned domain, e.g. `com.yourcompany.voxlingo`
- check Play Console acceptance on app creation

---

## F. Missing production backend readiness gate
Spec says backend is on local IP and user will handle hosting separately, but the plan still allows proceeding to preview/production builds with fallback local IP.

**Why this matters:** You can publish a build that is dead on arrival.

**Action:** Add a hard gate:
- no production build until `EXPO_PUBLIC_SERVER_URL` points to a reachable HTTPS production backend
- verify from device on mobile network, not just local Wi-Fi

---

## G. Missing Google/Firebase Android app registration steps
If Firebase or Google Sign-In is used, there may be required Android setup:
- register Android app in Firebase with package name
- add SHA-1 / SHA-256 fingerprints for auth providers
- configure OAuth redirect / Google Sign-In client IDs correctly

The docs mention Firebase credentials and Google OAuth client ID, but not Android-specific setup.

**Action:** Add steps:
- create Android app in Firebase for final package name
- add SHA-1/SHA-256 from EAS credentials / Play App Signing as needed
- verify Google Sign-In works in release build, not just dev

---

## H. Missing release testing matrix
The plan only says “test APK” and “test via internal track.”

**Action:** Add explicit test matrix:
- Android 13/14 physical device
- fresh install
- upgrade install
- denied permissions path
- offline/no network path
- slow network path
- background/foreground transitions
- app cold start after process kill
- camera/microphone permission revoked in settings
- production backend/API error handling

---

## I. Missing crash monitoring / analytics readiness
For first production release, you should have at least one of:
- Firebase Crashlytics
- Sentry
- Expo error reporting strategy

**Action:** Add crash monitoring before production rollout.

---

## J. Missing legal/policy checks for AI claims
The store description claims:
- “real-time”
- “50+ languages”
- “offline phrase caching”
- “cultural guide”

If any of these are incomplete or unreliable, this can be considered misleading listing content.

**Action:** Verify every listed feature exists in release build. Remove unsupported claims.

# 2) Security risks

## A. Misuse of `EXPO_PUBLIC_*`
This is the biggest issue.

Anything in `EXPO_PUBLIC_*` is **embedded in the client bundle** and is **not secret**.

The spec says:
> Store sensitive environment variables as EAS secrets

But then lists `EXPO_PUBLIC_FIREBASE_API_KEY`, `EXPO_PUBLIC_GOOGLE_CLIENT_ID`, etc.

That is misleading. EAS secrets can inject values at build time, but if the variable name is `EXPO_PUBLIC_*`, the value becomes public in the app.

**Action:** Update wording:
- `EXPO_PUBLIC_*` values are **public config**, not secrets
- only use them for values safe to expose in a mobile app
- do not put server-side secrets in Expo public env

---

## B. App should not contain server-side AI keys
The spec mentions rotating:
- Gemini API key
- OpenAI API key

These should generally **not** be shipped in the mobile app. They should live on your backend only.

If the app directly calls Gemini/OpenAI from the client, that is a major abuse/billing risk.

**Action:** Ensure:
- mobile app calls your backend only
- backend holds Gemini/OpenAI secrets
- app never contains OpenAI/Gemini secret keys

---

## C. Firebase API key handling is misunderstood
Firebase API keys are not secret in the same way as server keys, but they must still be restricted appropriately where possible and paired with proper Firebase security rules.

**Action:**
- keep Firebase config in app if needed
- enforce Firebase Auth / Firestore / Storage security rules
- do not rely on API key secrecy

---

## D. Google Maps API key must be restricted
If using Maps key in app:
- restrict to Android apps
- bind to package name + SHA-1 certificate fingerprint
- enable only required APIs

**Action:** Add explicit key restriction step in Google Cloud Console.

---

## E. Google OAuth client ID likely incomplete/wrong
For Android Google Sign-In, you usually need Android-specific OAuth setup and sometimes a web client ID depending on library flow. A generic `EXPO_PUBLIC_GOOGLE_CLIENT_ID` may be insufficient or wrong.

**Action:** Verify exact auth library in use and configure:
- Android client ID
- web client ID if required
- package name + SHA fingerprints

---

## F. Service account JSON on local disk
The plan says to store `google-service-account.json` locally and gitignore it. That’s okay for local use, but still risky.

**Action:** Better options:
- use CI secret storage if automating
- restrict service account permissions to minimum required
- rotate key if shared
- avoid broad project roles

---

## G. Git history exposure remediation incomplete
Rotating keys is necessary, but not sufficient.

**Action:** Also:
- audit where exposed keys were used
- revoke old tokens/clients
- check billing abuse
- consider rewriting git history only if repo is private and team agrees, but rotation is the real fix
- ensure `.env*` patterns are gitignored if not already

# 3) EAS config issues

## A. `appVersionSource: "remote"` + hardcoded `versionCode`
This can work, but the plan doesn’t explain how remote versioning interacts with local config. Teams often get confused when Play rejects duplicate version codes.

**Action:** Document expected behavior:
- `version` is marketing version
- Android `versionCode` is managed by EAS remote auto-increment for production
- verify current remote version before release

---

## B. `preview` as APK is fine, but internal testing on Play requires AAB
The plan mixes two concepts:
- preview APK for sideload testing
- internal testing track in Play with AAB

That’s okay, but should be stated more clearly.

**Action:** Clarify:
- `preview` APK = sideload only
- `production` AAB = Play upload

---

## C. `serviceAccountKeyPath` in repo root is brittle
This path assumes local execution from project root and local file presence.

**Action:** If keeping this:
- document exact working directory
- add validation step before `eas submit`
- consider using `--key` path explicitly in CI/manual workflows

---

## D. Missing build environment variables per profile
The plan creates EAS secrets globally, but doesn’t define environment per build profile.

**Action:** Consider:
```json
{
  "build": {
    "preview": {
      "env": {
        "APP_ENV": "preview"
      }
    },
    "production": {
      "env": {
        "APP_ENV": "production"
      }
    }
  }
}
```
Then use `APP_ENV` in `app.config.ts`.

---

## E. No `channel` / update strategy if using EAS Update later
Spec mentions OTA updates, but `eas.json` has no channels and app config has no update strategy.

**Action:** If OTA is planned, define:
- `channel` per profile
- `runtimeVersion`
- `updates.url` / Expo updates config as needed

Otherwise remove OTA section from current publishing scope.

---

## F. `eas init` may modify config unexpectedly
The plan notes it “may create/update app.json with extra.eas.projectId” but doesn’t account for dynamic config ownership.

**Action:** Prefer managing `extra.eas.projectId` in `app.config.ts` or at least verify post-init config diff.

# 4) Incorrect or risky Play Store requirements/statements

## A. “Internal testing — immediate availability, no review”
This is too absolute and can be misleading.

Google may still perform checks, and policy/app content declarations still matter. Internal testing is faster and lighter, but not “no review” in a guaranteed sense.

**Action:** Reword to:
- internal testing is usually the fastest distribution path and may have limited review friction, but policy declarations and app content setup are still required

---

## B. “Everyone” rating is not guaranteed
Likely yes, but not guaranteed until questionnaire is completed.

**Action:** Reword to:
- likely Everyone, subject to IARC responses

---

## C. Data Safety answers may be wrong depending on implementation
The docs say:
- audio/photos/location are collected and not stored
- not shared

But if data is sent to Google APIs, Firebase, or backend processors, “shared” and “collected” definitions must match Google’s Data Safety form precisely.

**Action:** Do not pre-commit to exact answers until data flow is mapped. Review:
- collected?
- shared?
- processed ephemerally?
- encrypted in transit?
- user can request deletion?
- required vs optional?

---

## D. Location declaration may be inaccurate
Spec says:
- privacy policy must cover device location for cultural context
- data safety says approximate location only
- app.json requests both fine and coarse location

This is inconsistent.

**Action:** Choose one:
- if only approximate location is needed, request coarse only
- if location is not implemented, remove location entirely from manifest and listing
- if fine location is truly needed, justify it clearly

---

## E. “No account required” may be false
Spec says no account required unless Firebase auth is mandatory. That needs verification.

**Action:** Confirm actual auth flow before filling Play declarations.

# 5) Build failure risks

## A. `app.config.ts` fallback to local IP
This is a major release risk:
```ts
serverUrl: process.env.EXPO_PUBLIC_SERVER_URL || "http://192.168.194.247:3001"
```

If the env var is missing in a cloud build, you may ship a build pointing to a dead local IP.

**Action:** Never use local IP fallback for production builds. Use environment-aware logic:
```ts
const isProd = process.env.APP_ENV === "production";
const serverUrl = process.env.EXPO_PUBLIC_SERVER_URL;

if (isProd && !serverUrl) {
  throw new Error("EXPO_PUBLIC_SERVER_URL is required for production builds");
}
```

---

## B. `expo-router` plugin may fail if dependency/config is missing
The plan hardcodes:
```json
"plugins": ["expo-router"]
```
without verifying the project already uses expo-router correctly.

**Action:** Confirm:
- `expo-router` is installed
- entry points are configured
- plugin is already present or actually needed

If not, don’t add it just because.

---

## C. `newArchEnabled: true` is unnecessary risk
The plan adds:
```json
"newArchEnabled": true
```
This was not in the spec and can introduce native compatibility issues.

**Action:** Do not enable New Architecture as part of publishing prep unless the app is already tested with it. Remove this change from the plan.

---

## D. Replacing full `app.json` is risky
The plan says “replace the full contents.” That can accidentally remove existing required config:
- plugins
- assetBundlePatterns
- experiments
- extra
- notification config
- deep link config
- intent filters
- jsEngine
- updates config

**Action:** Merge surgically, don’t replace wholesale.

---

## E. `platforms: ["android"]` may have side effects
This is usually okay, but removing iOS/web from config is not necessary for Android publishing and may affect dev tooling or future work.

**Action:** Unless there is a strong reason, leave platforms alone or verify no tooling depends on them.

---

## F. Asset file assumptions may fail build/submission
The plan assumes these files exist and are valid:
- `./assets/icon.png`
- `./assets/adaptive-icon.png`
- `./assets/splash-icon.png`

**Action:** Validate:
- files exist
- dimensions are appropriate
- icon has no alpha for Play listing asset
- adaptive icon foreground works on Android

---

## G. Missing check for cleartext HTTP usage
If any environment still points to `http://...`, Android release networking may fail or violate security expectations depending on implementation.

**Action:** Production backend must be HTTPS.

---

# 6) Gaps between spec and plan

## A. Spec says dynamic config should replace static config behavior; plan barely uses it
The plan puts almost everything in `app.json` and only adds `extra` in `app.config.ts`.

**Gap:** Dynamic config is not really being used for environment-based Android config or validation.

**Action:** Move environment-sensitive logic into `app.config.ts`, including:
- package if needed by env
- versioning strategy if needed
- server URL validation
- eas project ID if desired
- runtime/update config later

---

## B. Spec says “keep existing icon/splash/permissions config”; plan overwrites app.json
That’s contradictory.

**Action:** Audit current `app.json` and preserve existing valid settings rather than replacing whole file.

---

## C. Spec mentions privacy policy and key rotation as prerequisites; plan doesn’t gate on them
The plan proceeds to builds before confirming:
- privacy policy is hosted
- keys are rotated
- backend is production-ready

**Action:** Add explicit blockers before production build/submission.

---

## D. Spec mentions service account setup in detail; plan omits minimum Play permissions
The service account role “Service Account User” is not enough by itself for Play submission. The important part is Play Console API access permissions.

**Action:** Clarify:
- create service account in Google Cloud
- link in Play Console API access
- grant app-level Play permissions needed for release management
- “Service Account User” in GCP is not the main permission that matters for Play submission

---

## E. Spec mentions OTA updates; plan ignores required implementation
If `expo-updates` and `runtimeVersion` are not configured, the OTA section is aspirational only.

**Action:** Either:
- remove OTA from current scope, or
- add concrete implementation tasks

## Task 3 corrections
### Problems
- local IP fallback in production risk
- `extra` values are public, not secret
- no production env validation

### Recommended `app.config.ts` pattern
```ts
import { ExpoConfig, ConfigContext } from "expo/config";

export default ({ config }: ConfigContext): ExpoConfig => {
  const appEnv = process.env.APP_ENV ?? "development";
  const serverUrl = process.env.EXPO_PUBLIC_SERVER_URL;

  if (appEnv === "production" && !serverUrl) {
    throw new Error("EXPO_PUBLIC_SERVER_URL is required for production builds");
  }

  return {
    ...config,
    extra: {
      ...config.extra,
      appEnv,
      serverUrl: serverUrl ?? "http://localhost:3001",
      firebaseApiKey: process.env.EXPO_PUBLIC_FIREBASE_API_KEY ?? "",
      firebaseAuthDomain: process.env.EXPO_PUBLIC_FIREBASE_AUTH_DOMAIN ?? "",
      firebaseProjectId: process.env.EXPO_PUBLIC_FIREBASE_PROJECT_ID ?? "",
      googleClientId: process.env.EXPO_PUBLIC_GOOGLE_CLIENT_ID ?? "",
      googleMapsApiKey: process.env.EXPO_PUBLIC_GOOGLE_MAPS_API_KEY ?? "",
    },
  };
};
```
